Part 1 · 0:00 – 0:30

The ND-Array

contiguous, typed, fast

NumPy is installed — confirm the version

Sanity-check that NumPy imports and report its version. Every notebook in this course starts the same way.

  


  

    
List vs. ndarray — same data, different machine

    
Python lists store pointers to boxed Python objects; ndarrays store raw C numbers back-to-back in one contiguous typed buffer. That difference is the entire speed story.


    

      
      
      
    

    

  


  

    
Anatomy of an ndarray

    
Every ndarray is a header (shape, dtype, strides) pointing at a flat, contiguous block of typed memory.

    
    
The header is tiny. The buffer is the data. Slicing & reshape mostly just rewrite the header. By default NumPy fills that buffer row by row — the C-order layout (named after the C language).

  


  

    
View vs copy — same buffer or new buffer?

    
Because the buffer is separate from the header, NumPy has two ways to give you a "different array":

    

      

        
View

        
A new header pointing at the same buffer. Cheap (just metadata), and writes through to the original.

        
    
          
  • basic slicing — a[1:3]
    • 
          
  • reshape, ravel, transpose, .T
    • 
          
  • np.newaxis, expand_dims, squeeze
    • 
          
  • changing dtype with .view()
    • 
        

      

      

        
Copy

        
A brand-new buffer with its own bytes. Expensive (allocates + memcopies), independent of the original.

        
    
          
  • fancy / boolean indexing — a[[0,2]], a[a>0]
    • 
          
  • arithmetic — a + 1, a * 2
    • 
          
  • astype, flatten, .copy()
    • 
          
  • np.concatenate, np.stack
    • 
        

      

      

        
How to check

        
    
          
  • b.base is a — true if b is a view of a.
    • 
          
  • np.shares_memory(a, b) — true if they overlap.
    • 
          
  • arr.flags.owndata — false on a view.
    • 
        

        
When in doubt, .copy() explicitly. Mutating a view you forgot was a view is the #1 NumPy bug.

      

    

  


  

    
View vs copy — see the difference

    
Same source array a. The slice shares memory and writes through; .copy() and fancy indexing produce independent buffers.


    

      
      
      
    

    


  


  

    
How cells line up in RAM — C-order vs F-order

    
NumPy needs to flatten an n-dimensional logical shape into a 1-D buffer. Two conventions exist, named after the languages that popularised them:

    
    
      
  • C-order (row-major). The order used by the C / C++ languages: walk along the last axis fastest. NumPy's default.
    • 
      
  • F-order (column-major). The order used by Fortran (and inherited by MATLAB, R, Julia): walk along the first axis fastest.
    • 
    

    
Same logical matrix, different linear walk. The strides tuple in the header is what encodes which one you have.

  


  

    
C-order vs F-order — laying cells out in RAM

    
Same logical matrix, two possible linear orders. strides = how many bytes to jump per axis.

    
  


  

    
C-order vs F-order — why you'd care

    
    
      
  • Cache locality. CPUs love sequential reads. Walking rows of a C-order array (or columns of an F-order array) is cache-friendly — going the other way can be an order of magnitude slower.
    • 
      
  • Transpose is free. a.T moves no bytes — it just swaps strides, so a C-contiguous array becomes F-contiguous through its transpose.
    • 
      
  • When it bites. Some BLAS/LAPACK or SciPy routines silently copy your data if the layout is wrong. arr.flags shows which layout you actually have.
    • 
    

  


  

    
C-order vs F-order — see it

    
Same logical array, different memory walks. ravel() flattens in whatever order you ask for — that's the cleanest way to see the underlying buffer order.


    

      
      
      
    

    


  

  

    
NDArray header → memory block

    
A concrete (3, 3) float32 array: 4-byte elements, row stride 12, column stride 4.

    
    
The header carries shape + strides; the buffer carries bytes. Moving down a row in (3, 3) float32 = jump 12 bytes; moving across a column = jump 4 bytes.

  


  

    
Six recipes for creating arrays

    
99% of "make an array" reduces to one of these constructors: zeros, ones, full, arange, linspace, eye.


    

      
      
      
    

    

  


  

    
1-D, 2-D, 3-D — same ndarray, more axes

    

      

        
      

      

        
      

      

        
      

    

    
A 1-D row, a 2-D table, a 3-D stack of tables. ndim just counts the axes.

  


  

    
Random — use the modern API

    
np.random.seed is legacy. default_rng gives you a deterministic generator object — seed it once and pass it around.

    
    
      
  • No global state. Unlike np.random.rand() / np.random.seed(), it doesn't pollute the global RNG — no unintended side effects when libraries reseed behind your back.
    • 
      
  • Modern algorithm. Backed by PCG64 by default — faster, smaller state, and statistically stronger than the old Mersenne Twister.
    • 
    


    

      
      
      
    

    

  


  

    
dtypes and promotion

    
Adding two arrays promotes to the smallest dtype that holds both. Adding a float scalar always promotes to float64.


    

      
      
      
    

    

  


  

    
dtype memory footprint

    
Per-element cost in bytes. Multiply by size to get nbytes.

    
    
A million-element float64 array is 8 MB; the same data in float32 halves that — at the cost of precision.

  


  

    
astype — convert dtype, get a new array

    
astype returns a fresh array in the requested dtype. Going float→int truncates toward zero; no rounding.


    

      
      
      
    

    

  


  

    
Six attributes you always check

    
Every ndarray exposes shape, ndim, size, itemsize, nbytes, dtype. Inspecting these is the first move when a shape error bites.


    

      
      
      
    

    

  


  

    
See an array as a heatmap

    
Visualize an array as a heatmap to inspect its structure.


    

      
      
      
    

    

  






  

    Part 2 · 0:30 – 1:00
    
Shape manipulation

    
reshape, transpose, axes

  


  

    
Reshape changes strides, not data

    
Reshape is essentially free — it makes a view over the same buffer. ravel returns a 1-D view when possible; flatten always copies.


    

      
      
      
    

    

  


  

    
Reshape — row-major fill order

    
The buffer is unchanged; the header relabels how to walk it. NumPy fills the new shape row-by-row (C-order).

    
    
Same 12 numbers, same buffer. Only the strides change.

  


  

    
Strides — from [i, j] to a flat offset

    
The 2-D index [i, j] maps to a byte offset in one contiguous block: offset = i · strides[0] + j · strides[1].

    
    
Strides are the bytes to skip per axis step. That's why reshape and transpose are cheap — they rewrite strides, never the buffer.

  


  

    
Transpose — also a stride trick

    
Transposing a 2-D array is the same as .T; for higher dims you list the axis permutation explicitly.


    

      
      
      
    

    

  


  

    
Add and remove dimensions

    
You will use these constantly for broadcasting (next course) and for matching tensor shapes in deep learning. np.newaxis inserts a length-1 axis; squeeze drops them.

    
    
      
  • np.newaxis vs np.expand_dims — same mechanic, different syntax. Both insert a length-1 axis and return a view (no copy — only the shape/strides metadata changes). np.newaxis is an alias for None used inside slicing brackets and reads like the target shape (arr[:, np.newaxis]); np.expand_dims(arr, axis=k) is the function form and is the one to reach for when the axis is a variable or you need to insert several at once (axis=(0, 2)).
    • 
      
  • np.squeeze is the inverse. It drops every length-1 axis (or just the ones you name with axis=) — also a view, no data moved. Use it to clean up stray singleton axes left over from reductions like arr.sum(axis=1, keepdims=True) or model outputs shaped (N, 1).
    • 
    


    

      
      
      
    

    

  


  

    
Axis convention — what gets collapsed

    
The axis= argument names the axis that disappears after the reduction.

    
    
Mnemonic: axis=k removes the k-th entry of the shape tuple.

  


  

    
Axis convention — in code

    
Same 3×4 grid from the diagram. Watch which entry of the shape tuple disappears: axis=0 kills the 3, axis=1 kills the 4. keepdims=True leaves a length-1 placeholder behind — handy for broadcasting the reduced values back against the original.

Vectorize & ufuncs

  


  

    
Loop vs vectorized — the picture

    
The Python loop walks one cell at a time. NumPy hands the whole array to a C kernel that processes many cells per CPU instruction.

    
    
Same math, two implementations — the runnable on the next slide times them both.

  


  

    
The loop you stop writing

    
Same math, two implementations. The vectorized fast is typically 50–100× quicker than the Python loop — and shorter to read. (We time with time.perf_counter in place of Jupyter's %timeit.)


    

      
      
      
    

    

  


  

    
Timing sin(x)² — loop vs vectorized

    
Same result; vastly different speed. Compute sin(x)**2 both ways, time each, and plot the resulting curve with both timings in the title.


    

      
      
      
    

    

  


  

    
Element-wise arithmetic on paired arrays

    
If shapes match, every operator applies element-by-element. If they don't, broadcasting (next part) makes them match.


    

      
      
      
    

    

  


  

    
Universal functions — math everywhere

    
A "ufunc" operates element-wise on any-shape array. np.maximum(a, 0) is ReLU in one line; np.clip bounds values to a range.


    

      
      
      
    

    

  


  

    
Reductions — and the axis mnemonic

    
Mnemonic: axis=k means "the axis k disappears." mean() with no axis collapses everything to a scalar.


    

      
      
      
    

    

  


  

    
Reduction along an axis — that axis disappears

    
Sum across rows (axis=0) collapses to a single row; sum across columns (axis=1) collapses to a single column.

    
    
Mnemonic: axis=k means axis k disappears. Use keepdims=True if you need to preserve rank for later broadcasting.

  


  

    
Reducing along specific axes

    
On a 3-D array, sum(axis=0) collapses the leading axis; sum(axis=(0, 2)) collapses two axes at once and leaves only axis 1.

Broadcasting

    
mixed shapes, no loops

  


  

    
The rule, in one sentence

    
Two shapes are compatible if, aligned right-to-left, every pair of dimensions is equal or one of them is 1.

    
A length-1 axis is stretched to match. A missing axis is treated as length 1.

  


  

    
Broadcasting rules, in pictures

    
    
Equal, or one of them is 1, axis by axis — right-aligned. That single rule covers every legal broadcast.

  


  

    
Scalar broadcast

    
A scalar has shape () — it broadcasts to every cell of the array. No loop, no copy of the scalar.


    

      
      
      
    

    

  


  

    
Row vector across rows: (2,3) + (3,)

    
A length-3 vector aligns with the trailing axis of a (2, 3) matrix and is reused across every row.


    

      
      
      
    

    

  


  

    
Column vector across columns: (2,3) + (2,1)

    
A column vector shape (2, 1) stretches across the columns of a (2, 3) matrix.


    

      
      
      
    

    

  


  

    
Two compatible shapes: (3,1) + (1,4) → (3,4)

    
When both operands have a length-1 axis in different places, broadcasting expands both — here producing a (3, 4) outer-sum.


    

      
      
      
    

    

  


  

    
Broadcasting, visualized as a heatmap

    
Outer addition via broadcasting, as a heatmap. Each cell (i, j) is a[i] + b[j].

Linear algebra & stacking

  


  

    
Matmul shape rule — inner dims must match

    
Why (m, k) @ (k, n) → (m, n): the shared k contracts away.

    
    
If the inner dimension disagrees, NumPy raises ValueError: shapes ... not aligned. Print shapes to debug.

  


  

    
Matrix multiplication — the @ operator

    
@ calls np.matmul (equivalent to .dot for 2-D arrays). Never use * for matrix multiplication — that's element-wise.


    

      
      
      
    

    

  


  

    
Solve a linear system — Ax = b

    
Always prefer np.linalg.solve over inv(A) @ b — it uses LU factorization, so it's faster and numerically more stable.

    Big arrays
    
When the array doesn't fit

    
An ndarray needs one contiguous block of RAM. Asking for np.zeros((50_000, 50_000)) in float64 wants 20 GB in one piece. Even with free RAM, fragmentation can make that allocation fail with MemoryError.

  


  

    
The size, in your head

    
    
      
  • bytes = prod(shape) × dtype.itemsize. Always do this math before allocating.
    • 
      
  • float64 = 8 B/cell, float32 = 4 B, int8/bool = 1 B.
    • 
      
  • A (10_000, 10_000) float64 matrix is 800 MB — and you usually need a copy for the next step.
    • 
    

    
Four escape hatches: downcast dtype, views not copies, chunked generators, memory-mapped files.

  


  

    
1) Downcast the dtype

    
Same shape, a quarter of the bytes. Pick the smallest dtype the data actually needs.


    

      
      
      
    

    


  


  

    
2) Views, not copies

    
Slicing and reshape share memory. .copy(), fancy indexing, and most arithmetic allocate. Know which is which.


    

      
      
      
    

    


  


  

    
3) Chunked generators — process, don't load

    
When the full array won't fit, stream it. A generator yields one chunk at a time; only the chunk lives in RAM.


    

      
      
      
    

    


  


  

    
4) np.memmap — array backed by a file

    
A memmap behaves like an ndarray but lives on disk. The OS pages in only the slices you actually touch — you can index a 100 GB array on a laptop.


    

      
      
      
    

    


  


  

    
Choosing the right hatch

    
    
      
  • Fits in RAM but barely? Downcast dtype, prefer views.
    • 
      
  • One pass over many rows? Generator + running stats.
    • 
      
  • Random access on a huge array? np.memmap (or HDF5/Zarr).
    • 
      
  • Bigger than one machine? Dask, Ray, or a database — out of scope here.
    • 
    

    
MemoryError is rarely "you're out of RAM" — it's "you asked for one giant contiguous block." Reshape the problem first, buy more RAM last.
